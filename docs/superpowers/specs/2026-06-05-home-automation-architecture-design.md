# Spec 1 — Home Automation Architecture + State-of-World Doc Rewrite

> Status: **Approved** (2026-06-05). Brainstormed + crit-reviewed (2 rounds). Next: implementation plan via writing-plans.

## Context

The smart home (HA + Node-RED + Mosquitto + ESPHome on `blacky`, repo `HomeAutomationSetup`)
has reached a milestone: the **sensing and climate-control layers are live** (WAQI outdoor,
2× Qingping, Xiaomi purifiers, Broadlink RF fans, LocalTuya boiler, Met.no weather, Telegram
bot), but the **automation layer is empty** — `automations.yaml`, `scripts.yaml`,
`scenes.yaml` all `[]`, Node-RED has zero flows.

Before writing any automation, decide the *engine + workflow* so future automations stay
diffable, reviewable, low-context-overload, and self-documenting. Guiding philosophy:
**when unsure, notify with an easy call-to-action over automating fragile edge cases**
(e.g. blinds-down is a CTA, not a schedule).

This spec delivers **only** the architecture/tooling + a refreshed source-of-truth doc.
The climate flagship automation and other sub-projects are **deferred to their own
sequenced specs**.

### Decisions locked
- **Engine = hybrid:** HA YAML + blueprints for simple/trivial automations; Node-RED reserved
  for genuinely complex/stateful/multi-input flows. (NR currently 0 flows → scaffolding
  deferred to first complex flow; architecture decided + documented now.)
- **Node-RED file split:** `node-red-contrib-flow-manager` → per-domain files; subflows for
  reusable patterns; file-backed persistent global context.
- **Diff control:** `jq` normalizer (stable key sort + round/strip canvas coords) + externalized
  Function-node JS for ESLint.
- **Self-documenting ToC:** `docs/automations.md` with an autogenerated table block, driven by
  a required per-automation `meta` annotation. Enforced by commit hook.
- **CTA channel = Telegram** (already live; two-way inline buttons via `telegram_callback`;
  reaches Yossi remotely with no HA exposure). HA companion app revisited after remote-access spec.
- **Scope = architecture + doc only.**

## Part A — Automation engine model (documented conventions)

Add `docs/automation-architecture.md` (the "how we build automations" guide):

1. **Engine-choice rubric**
   - Single trigger → single action/notification, no shared state → **HA YAML** (`automations.yaml`)
     or a **blueprint** instance.
   - ≥2 inputs fused, timers/debounce, state machine, or cross-domain coordination → **Node-RED** flow.
2. **Naming:** every automation has a **globally unique alias** (enforced by the hook),
   **domain-prefixed** (`climate-…`, `boiler-…`, `presence-…`). kebab-slug or human-readable both
   fine — pick one and stay consistent. The alias is the ToC key, so it must stay unique.
3. **Required `meta` annotation** on every automation (source of ToC truth):
   - HA YAML: a comment line directly above each block — `# meta: intent="…"; waf=low|med|high; mode=auto|cta`
   - Node-RED: the node/tab `info` field carrying the same `intent`/`waf`/`mode` keys.
4. **CTA-over-automation principle** documented with the Telegram primitive (Part D).

## Part B — Node-RED architecture (decided now, scaffolded in flagship spec)

Document in `docs/automation-architecture.md`; implement when first complex flow lands:
- Install **`node-red-contrib-flow-manager`** → per-domain flow files under the NR data dir
  (`~/homeassistant/nodered/data/flows/`), e.g. `climate.json`, `presence.json`.
  *(Verify module name/maintenance before install.)*
- **Subflows** for reusable patterns (Telegram-CTA subflow mirrors the HA script).
- **Persistent global context:** set `contextStorage` (file store) in NR `settings.js` for shared
  runtime vars (`presence`, `outdoor_aqi`, `co2`).
- **Externalize Function-node JS:** enable `functionExternalModules` + keep heavy logic in versioned
  `nodered/data/lib/*.js` → lints with ESLint.
- **Git tracking:** track `flows/*.json`, `settings.js`, `lib/`; **gitignore** `flows_cred.json`,
  `.config.*.json`, `.sessions.json`, `node_modules/`, `lib/node_*`.

## Part C — Diff-noise control (normalizer)

- `scripts/normalize_nodered.sh` — `jq` filter: stable key sort + round/strip node `x`/`y` canvas
  coords (and other volatile fields) on each `flows/*.json`. Run by the commit hook so GUI-driven
  layout churn doesn't pollute logic diffs.
- ESLint config (`.eslintrc`) scoped to `nodered/data/lib/*.js`.

## Part D — Telegram call-to-action primitive

Reusable CTA building block (the existing `confirmable_notification.yaml` blueprint is
`mobile_app`-only; Telegram needs its own):
- Add **`telegram-confirmable`** script in `config/scripts.yaml` (or a script blueprint): send
  `telegram_bot.send_message` with an `inline_keyboard`, `wait_for_trigger` on the
  `telegram_callback` event matching this context, then run confirm/dismiss actions and
  `answer_callback_query`/edit the message.
- Document the pattern in `docs/automation-architecture.md` as the default for any "unsure → ask"
  decision (blinds-down, window-vs-AC, etc. — consumed by later specs).

## Part E — Self-documenting ToC + enforcement commit hook

1. **`docs/automations.md`** — human prose/design-notes section + an
   `<!-- AUTOGEN:automations START -->…<!-- AUTOGEN:automations END -->` block.
2. **`scripts/gen_automations_toc.py`** (runs in existing pyenv `homeautomation`):
   - Parse `config/automations.yaml` + `config/scripts.yaml` for `alias` + `# meta:` annotations.
   - Parse `nodered/data/flows/*.json` for automation tabs/nodes + `info` `meta` keys.
   - Emit table: `name | engine | file | trigger | devices | intent | waf | mode` into the AUTOGEN block.
3. **Versioned git hook** via `core.hooksPath=scripts/git-hooks` (no extra dependency):
   `scripts/git-hooks/pre-commit` runs, in order:
   - `yamllint` on `config/*.yaml`; ESLint on `nodered/data/lib/*.js`.
   - `normalize_nodered.sh` on staged `flows/*.json`.
   - **Validate** every automation carries a `meta` annotation; **fail on missing meta or duplicate names**.
   - Regenerate the ToC AUTOGEN block; `git add docs/automations.md`.
   - One-time setup line in README: `git config core.hooksPath scripts/git-hooks`.
4. **HA config validation** (needs the HA container, so not in pre-commit): a `make check` /
   documented `docker compose exec homeassistant python -m homeassistant --script check_config -c /config`,
   run on blacky / as a pre-push.
