# CI — GitHub PR Gating + Branch Protection — Design

> Move the repo's currently local-only, bypassable checks into enforced GitHub
> Actions CI so a PR must be green before it can merge, and `origin/main` stays
> trustworthy for blacky to pull. This is the **CI** half; **CD** (blacky safe
> auto-deploy loop, where HA `check_config` lives) is a separate later spec that
> depends on backlog #10 (config ownership).

## Problem

The repo already has real checks, but nothing enforces them:

- `scripts/git-hooks/pre-commit` (yamllint + `gen_automations_toc.py --check` + ToC
  regen) is **local-only and bypassable** with `git commit --no-verify` (used twice
  during the #1A doc commits).
- The Makefile (`make lint` / `toc` / `test` / `check`) and the `tests/` pytest suite
  run only when someone remembers to run them.
- blacky pulls `origin/main` and runs nothing on the source — a lint-clean-but-broken
  or stale-ToC commit can reach `main` unnoticed.

We want: a PR cannot merge unless the fast checks pass, and `main` only ever advances
through that gate.

## Scope

**In:** a GitHub Actions workflow that runs the fast, dependency-light checks on every
PR to `main`, as **parallel independent jobs**; branch protection on `main` requiring
those checks + a PR (squash-merge, linear history); README documentation of the flow.

**Out (own specs / later):**
- **CD** — blacky safe auto-deploy loop (fetch → `check_config` in the real HA container
  → reload/rollback → Telegram alert). HA `check_config` belongs there, not in CI: blacky
  is the only place the HACS integrations (`config/custom_components/*/` is gitignored) and
  the real `secrets.yaml` exist. CD depends on backlog **#10** (chown `config/` off root so
  an unattended `git pull` works).
- Required PR **approvals** — the repo is solo; GitHub blocks self-approval, so requiring
  one would deadlock merges. Status checks are the gate, not human review.
- Auto-merge — explicitly not enabled; green makes a PR *eligible*, a human still clicks
  merge, so half-baked branches never merge themselves.

## Cost

`yossisolomon/HomeAutomationSetup` is **public** → GitHub Actions minutes are free and
unlimited. Parallel jobs cost no more in practice. (Even private: the fast checks are
~1–2 min, far under the 2,000 free min/mo.)

## Design

### Workflow `.github/workflows/ci.yml`

Trigger: `pull_request` to `main` (events `opened`, `synchronize`, `reopened`) — so it
re-runs on every push to a PR branch. (No `push` trigger needed: `main` only advances via
squash-merged PRs that already passed, and direct push is blocked.)

**Parallel jobs** (the checks are mutually independent, so each is its own job and they run
concurrently; wall-clock ≈ the slowest job). Every job runs on `ubuntu-latest` and starts
with the same setup so they're self-contained:

1. `actions/checkout@v4`
2. `actions/setup-python@v5` with `python-version: "3.11"` (matches the local `homeautomation`
   pyenv env, Python 3.11.4) and `cache: pip`.
3. `pip install -r requirements-dev.txt` (PyYAML, pytest, yamllint).

Then each job runs one concern, **reusing the existing Makefile targets so CI ≡ local
(no drift)**:

| Job | Command | Catches |
|-----|---------|---------|
| `lint` | `make lint` (`yamllint config/`) | YAML style/syntax regressions |
| `toc` | `python scripts/gen_automations_toc.py --check` **then** `python scripts/gen_automations_toc.py && git diff --exit-code docs/automations.md` | missing `# meta:` / duplicate alias **and** a stale ToC left behind by a `--no-verify` commit |
| `pytest` | `python -m pytest tests/ -v` | the Python tooling test suite |
| `normalizer` | `bash tests/test_normalize_nodered.sh` | the Node-RED flow normalizer (`jq`, preinstalled on `ubuntu-latest`) |

The `toc` job's freshness step is the one piece not in the local Makefile-as-gate today; it
makes the autogenerated `docs/automations.md` self-enforcing even if the pre-commit hook was
bypassed.

### Branch protection on `main`

Configured via `gh api` (scripted in the plan; re-runnable) and documented in the README:

- **Require status checks to pass before merging:** `lint`, `toc`, `pytest`, `normalizer`
  (all four jobs). "Require branches to be up to date before merging" on.
- **Require a pull request before merging.** No required approvals (solo).
- **Block direct pushes** to `main` (enforced for admins too, optionally — see Open question).
- **Squash-merge only** + **require linear history** → one clean commit per feature on `main`
  (mirrors the manual squash done for #1A), keeping `main` quiet.

### Bootstrap order (chicken-and-egg)

Branch protection can only require a check that GitHub has seen run at least once, and the
workflow only exists once merged. Sequence:

1. Land `ci.yml` via its own PR from `feat/ci-github-gating`. For a same-repo branch,
   `pull_request` workflows run **from the PR branch**, so this introducing PR is itself
   checked by the new jobs — verifying them before they guard anything.
2. Merge that PR (squash).
3. *Then* enable branch protection requiring the four now-existing checks.

The README records this so a from-scratch rebuild re-applies protection in the right order.

## Feature flow (documented in README)

```
git checkout -b feat/<x>      # branch off main
# ...work, commit (local pre-commit hook = fast feedback)...
git push -u origin feat/<x>   # opens/updates the PR; CI runs the 4 jobs in parallel
gh pr create / gh pr view     # PR; merge button stays disabled until all 4 are green
# merge (squash) only when green AND you're satisfied — green != auto-merge
```

The local `scripts/git-hooks/pre-commit` stays as-is (fast pre-push feedback); CI is the
enforced backstop. They run the same `make`/script commands, so they agree.

## Files

- `.github/workflows/ci.yml` (new) — the four parallel-job workflow.
- `README.md` (modify) — the feature flow above + a "branch protection setup" note with the
  `gh api` command and the bootstrap order.
- `docs/state-of-world.md` (modify) — add CI done / CD (#10-dependent) to the backlog.

No changes to the existing hook, Makefile, or `tests/` — CI reuses them as-is.

## Validation

1. Open the introducing PR; confirm all four jobs appear and run **in parallel** and pass.
2. Push a commit that breaks one check each (a yamllint error; an automation with no
   `# meta:`; a hand-edited stale `docs/automations.md`; a failing assertion) on a throwaway
   branch → confirm the matching job (and only it where independent) goes red and the PR
   merge button is blocked.
3. After enabling protection: attempt a direct `git push origin main` → rejected.
4. Confirm a green PR still requires a manual merge click (no auto-merge).
5. Confirm squash-merge produces a single commit on `main`.

## Out of scope (later specs)

- **CD** — blacky auto-deploy loop + HA `check_config` + Telegram alerts (depends on #10).
- HA `check_config` in CI (rejected: needs HACS bootstrap + secrets stub; lives on blacky).
- Required reviewers / CODEOWNERS (solo repo).
- Release tagging, changelog automation, dependabot.

## Open question (decide at implementation)

**Enforce branch protection on admins too?** "Include administrators" makes the rules apply
to you as well (can't bypass with a direct push) — maximally honest, but means even a trivial
typo fix needs a PR. Recommendation: **on** (the whole point is a trustworthy `main`; emergency
bypass is one toggle away if ever needed).
