# HACS Plugin Tracking Policy тАФ Design

## Context

The `HomeAutomationSetup` repo wants git to be the source of truth for the
`blacky` Home Assistant config, but HACS-installed custom components are tracked
inconsistently тАФ some fully committed, some partially, some not at all:

| Component | tracked / on-disk | state |
|-----------|-------------------|-------|
| `ar_smart_ir` | 0 / 526 | untracked |
| `qingping_cgs1` | 0 / 24 | untracked |
| `localtuya` | 24 / 42 | code tracked, rest = pycache |
| `xiaomi_home` | 69 / 104 | code tracked, rest = pycache |
| `hacs` | 2258 / 2309 | code tracked, rest = pycache |

This generalizes backlog item #11 (was: just `ar_smart_ir` + `qingping_cgs1`) to
**all** HACS plugins, present and future.

### Verified facts (read-only inventory on blacky)

- **No plugin stores user-generated data inside its own dir.** All on-disk content
  is pristine upstream: integration code plus shipped reference data (e.g.
  `ar_smart_ir/codes/**` IR databases, `xiaomi_home/miot/specs/**`, locale files).
  `git status` shows zero local modifications to tracked plugin files; the only
  untracked non-pycache files are upstream files that were simply never committed.
- **Runtime / user state lives in `.storage/`** (config entries, auth tokens,
  device pairings, learned IR), not in the component dirs. `.storage/` is already
  gitignored (it holds secrets) and is HA-managed тАФ out of scope here.
- **HACS's own install list** lives in `.storage/hacs.repositories` (root-owned,
  ~3 MB, 3122 known repos). It is the authoritative record of what is installed.
- **5 installed integrations:**

  | full_name | category | domain |
  |-----------|----------|--------|
  | `hacs/integration` | integration | `hacs` |
  | `rospogrigio/localtuya` | integration | `localtuya` |
  | `mash2k3/qingping_cgs1` | integration | `qingping_cgs1` |
  | `marsh4200/ar_smart_ir` | integration | `ar_smart_ir` |
  | `XiaoMi/ha_xiaomi_home` | integration | `xiaomi_home` |

  Version pin source per repo: `version_installed` (release tag, when present) else
  `installed_commit` (SHA). The HACS entry exposes both, plus `category`, `domain`,
  `full_name`.

## Decisions locked during brainstorming

- **Policy = manifest-only**, applied uniformly to every HACS plugin. Gitignore the
  plugin code dirs; track a generated manifest instead of ~2350 vendored files.
- **Manifest-only is safe for code only.** If any plugin ever stored data we depend
  on inside its own dir, that data would need separate handling тАФ verified none do
  today; the shipped reference DBs are reproducible by pinning the repo version.
- **Reinstall mechanism = HACS-managed restore.** The manifest is the reference you
  re-add from; on rebuild you reinstall through HACS so its update UI keeps working.
  (Rejected: git-clone-at-ref bootstrap тАФ deterministic but loses HACS update
  tracking.)
- **Manifest location/name:** `hacs-manifest.yaml` at the repo root.
- **`.storage/` stays gitignored** (secrets); not a backup mechanism. Backing up
  `.storage` is backlog #10, separate.

## Architecture

Three tracked artifacts plus a cleanup:

1. **`hacs-manifest.yaml`** (repo root, generated) тАФ the committed record of installed
   HACS plugins.
2. **`scripts/gen_hacs_manifest.py`** тАФ generator, runs **on blacky** (only place
   `.storage/hacs.repositories` exists), reads the HACS store, writes the manifest.
3. **Rebuild runbook** тАФ documented restore steps (in the manifest header comment and
   referenced from `README.md` / `docs/state-of-world.md`).
4. **Untrack + gitignore cleanup** тАФ stop tracking the plugin code dirs.

### Component 1 тАФ `hacs-manifest.yaml`

Generated data file. Shape:

```yaml
# HACS plugin manifest тАФ generated by scripts/gen_hacs_manifest.py on blacky.
# Do not hand-edit. Regenerate after installing/updating a HACS plugin, then commit.
#
# REBUILD RUNBOOK (HACS-managed restore):
#   1. Install HACS (hacs/integration) per the official docs; restart HA.
#   2. For each plugin below:
#        - source: custom-repo  -> HACS > Integrations > (тЛо) Custom repositories,
#                                   add `repo` URL with the listed `category`, then
#                                   install and select `version`.
#        - source: default-store -> search the listed `name`/`full_name` in HACS and
#                                   install at `version`.
#   3. Restart HA. Re-link config entries (tokens/pairings live in .storage, not git).
plugins:
  - full_name: rospogrigio/localtuya
    domain: localtuya
    category: integration
    version: "<tag-or-commit:SHA>"
    source: default-store        # or custom-repo
  # ... one entry per installed plugin (hacs/integration included)
```

Field rules:
- `version`: the release tag if `version_installed` is set, else `commit:<installed_commit>`.
- `source`: `custom-repo` if the repo is not in the HACS default store (must be added
  by URL on restore), else `default-store`. The generator cannot perfectly detect
  default-store membership offline, so it defaults unknown repos to `custom-repo`
  (the safe, always-works path) and notes this; the user may correct entries.
- `repo`: `https://github.com/<full_name>` derived for convenience.

### Component 2 тАФ `scripts/gen_hacs_manifest.py`

- Reads `~/homeassistant/config/.storage/hacs.repositories` (path is the HA config
  dir on blacky; requires sudo as the file is root-owned).
- Filters to entries where `installed` is truthy or `version_installed` is set.
- Emits `hacs-manifest.yaml` sorted by `full_name` for stable diffs.
- Pure transform + file write; no network. Run as
  `sudo python3 scripts/gen_hacs_manifest.py [--storage <path>] [--out <path>]`.
- Defaults: storage = `~/homeassistant/config/.storage/hacs.repositories` resolved
  against the invoking user's home, out = repo-root `hacs-manifest.yaml`.
- Unit-tested against a fixture slice of `hacs.repositories` (a few installed + many
  not-installed entries) asserting: only installed repos emitted, tag-vs-commit pin
  selection, stable sort, derived `repo` URL.

### Component 3 тАФ Untrack + gitignore cleanup

- `.gitignore`: add `config/custom_components/*/` (ignore every plugin dir; the
  manifest is at repo root so it is unaffected). Keep `config/custom_components/`
  itself present if anything non-plugin lives there (nothing does today).
- `git rm -r --cached config/custom_components/<each currently-tracked dir>` to stop
  tracking the ~2350 vendored files without deleting them from blacky's disk.
- Single dedicated cleanup commit, separate from the generator/manifest commit.

## Data flow

```
blacky .storage/hacs.repositories  --(gen_hacs_manifest.py, sudo)-->  hacs-manifest.yaml
                                                                            |
                                                          git commit + pull to Mac
                                                                            |
                                            rebuild: human reads runbook -> HACS reinstall
```

## Error handling

- Missing/unreadable storage file тЖТ generator exits non-zero with the resolved path
  and a hint to run with sudo / pass `--storage`.
- Malformed JSON тЖТ caught, reported with the path; no partial manifest written
  (write to a temp file, then atomic rename).
- Zero installed plugins found тЖТ warn (likely wrong path) and exit non-zero rather
  than writing an empty manifest.

## Testing / verification

1. `gen_hacs_manifest.py` unit tests pass (fixture-driven).
2. Run generator on blacky тЖТ `hacs-manifest.yaml` lists exactly the 5 installed
   integrations with plausible version pins.
3. After `git rm --cached` + gitignore: `git ls-files config/custom_components/` is
   empty; `git status` shows the dirs ignored (not "deleted"); HA still starts and
   loads all integrations (the files are still on disk).
4. `git check-ignore config/custom_components/ar_smart_ir/climate.py` тЖТ ignored.

## Out of scope (own backlog items)

- `.storage` backup / blacky config ownership + privilege hardening (#10).
- Committing the z2m config edit / broader blacky git drift (#11's sibling).
- `scripts/` subject-dir reorg (#12).
- Any automated (non-HACS) reinstaller тАФ explicitly rejected in favor of HACS-managed
  restore.
